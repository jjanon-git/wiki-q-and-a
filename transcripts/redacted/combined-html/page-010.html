# Plan: `search_wikipedia` and tool implementation

Scope: the Wikipedia retrieval layer and the tool surface the agent sees. Not the agent loop, not the system prompt, not the eval harness.

## What "the tool" actually is

Two distinct things, easy to conflate:

1. **The Python function** `search_wikipedia(query: str) -> list[SearchResult]` — pure code. Hits MediaWiki, parses, returns structured results.
2. **The tool definition** — a JSON schema (name, description, input_schema) registered with the Anthropic API. The model never sees the function; it sees the schema and the stringified result we return as `tool_result` content.

The Python function is testable code. The tool definition is a prompt — its description and parameter docs steer model behavior.

## MediaWiki strategy

Use the action API with a generator to do search + extract in one HTTP call:

```
GET https://en.wikipedia.org/w/api.php
  ?action=query
  &format=json
  &generator=search
  &gsrsearch=<query>
  &gsrlimit=3
  &prop=extracts
  &exintro=true
  &explaintext=true
  &exchars=500
  &redirects=1
```

Returns the top-3 search hits as page objects, each with a plain-text lead-section extract truncated to ~500 chars, redirects followed. One round trip, no per-result fetch.

Alternatives considered:
- Two-step (search → per-title summary fetch via REST `/page/summary`): more requests, more failure modes, no quality gain for v1.
- Local Wikipedia dump or vector index: vastly out of scope for "lightweight v1."
- `wikipedia` PyPI package: thin wrapper, opaque error handling, brings its own opinions. Direct API call is not much more code and is fully ours.

## Return shape

```python
@dataclass(frozen=True)
class SearchResult:
    title: str
    url: str
    extract: str
    page_id: int
```

Internal API:

```python
def search_wikipedia(
    query: str,
    *,
    limit: int = 3,
    extract_chars: int = 500,
) -> list[SearchResult]: ...

def format_results_for_model(results: list[SearchResult]) -> str: ...
```

The agent dispatcher calls `search_wikipedia(query)` and passes `format_results_for_model(...)` as the `tool_result` content (a single string).

Why string vs JSON for the tool result: strings read more naturally for the model and waste fewer tokens on structural noise. We can revisit if the model starts hallucinating fields it expects from a structured response.

## Format for the model

```
Found 3 results for "<query>":

[1] <Title>
URL: https://en.wikipedia.org/wiki/<Title>
<extract...>

[2] ...
```

If zero results: `No Wikipedia results found for "<query>". Try a different query.`
If API error: `Wikipedia search failed: <reason>. The model should try again or answer without this tool.` — passed back as tool_result so the model can recover instead of crashing the loop.

## Tool definition (the prompt-engineering surface)

```python
SEARCH_WIKIPEDIA_TOOL = {
    "name": "search_wikipedia",
    "description": (
        "Search English Wikipedia and return the top results with short "
        "extracts. Use when you need factual information you do not "
        "reliably know, want to verify a claim, or need to ground an "
        "answer in a citable source.\n\n"
        "Returns up to 3 results. Each result includes a title, URL, "
        "and the lead section truncated to ~500 characters. If the "
        "first result does not answer the question, you may call this "
        "tool again with a refined query."
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": (
                    "Search terms — typically 2-6 keywords identifying "
                    "the entity, event, or concept. Avoid passing the "
                    "user's full question; favor specific noun phrases. "
                    "Example: 'Treaty of Versailles signatories' rather "
                    "than 'who signed the Treaty of Versailles?'."
                ),
            }
        },
        "required": ["query"],
    },
}
```

This is a v1 draft. The descriptions are themselves a prompt and likely to iterate based on eval results — for instance, if the model passes full questions as queries we'll tighten the parameter doc; if it never re-searches when results are weak we'll strengthen the "you may call again" sentence.

## Edge cases handled in v1

| Case | Behavior |
|---|---|
| Zero results | Return empty list; formatter emits "No results found" message |
| Disambiguation page in results | Pass through — the extract typically lists candidates, model handles |
| Empty `extract` field for a hit | Include title + URL with `<no extract available>` |
| HTTP 4xx/5xx | Raise `WikipediaSearchError`; agent catches and returns error string as tool_result |
| Network timeout | `httpx.TimeoutException` → wrapped in `WikipediaSearchError` |
| Unicode / special chars in query | `httpx` URL-encodes automatically |
| Query > some absurd length | No explicit limit in v1; MediaWiki will reject; we surface the error |

Explicitly **not** in v1: retry/backoff, on-disk caching, language selection, redirect-trace exposure, image/media. Add if eval evidence demands.

## HTTP client

`httpx` (sync). Reasons: typed, modern, mockable via `respx`. Sync because the agent is sequential. Easy to migrate to async later for parallel eval runs if needed.

User-Agent header: Wikipedia's policy expects an identifying UA. Set something like `wiki-qa-takehome/0.1 ([REDACTED_EMAIL])` — failure to do this can get requests rate-limited or blocked.

## Test plan (TDD)

Order matches red-green-refactor — each test written first, fails, then minimal impl.

**Unit tests** (`tests/unit/test_wikipedia.py`, mocked with `respx`):

1. `test_format_results_for_model_renders_expected_layout` — pure function; easiest entry point.
2. `test_format_results_for_model_empty_list` — emits "no results" message.
3. `test_search_wikipedia_parses_top_n_results` — mock MediaWiki JSON → assert SearchResult fields populated correctly.
4. `test_search_wikipedia_zero_results` — empty `pages` from API → empty list, no exception.
5. `test_search_wikipedia_respects_limit_and_extract_chars` — assert query string contains `gsrlimit=N` and `exchars=M`.
6. `test_search_wikipedia_url_encodes_query` — query with spaces/unicode/quotes → URL is correct.
7. `test_search_wikipedia_raises_on_http_5xx` — mock 500 → `WikipediaSearchError`.
8. `test_search_wikipedia_raises_on_timeout` — mock timeout → `WikipediaSearchError`.
9. `test_search_wikipedia_handles_missing_extract_field` — page with no extract → `SearchResult(extract="<no extract available>")`.
10. `test_search_wikipedia_sets_user_agent` — assert outbound request has the UA header.

**Tool-definition tests** (`tests/unit/test_tools.py`):

11. `test_search_wikipedia_tool_schema_required_fields` — name, description, input_schema present; `query` is required string. Cheap regression guard.

**Integration test** (`tests/integration/test_wikipedia_live.py`, `@pytest.mark.integration`, skipped by default):

12. `test_live_search_returns_results_for_stable_query` — real call for "Albert Einstein"; assert ≥1 result with non-empty extract. Run once to capture a real response, save as a fixture for the unit tests so they reflect actual API shape; re-run when we suspect the API changed.

## What this plan deliberately leaves open

- **N=3 and 500 chars are guesses.** They're starting points. Real values come from watching eval failures: too few results → bump N; too little context → bump chars; too much context bloat → cut.
- **Whether to add a separate `fetch_wikipedia_article(title)` tool.** Decision deferred until we see what the single-tool agent struggles with. If it repeatedly issues near-identical re-searches because extracts are too thin, that's the signal to add the second tool.
- **Whether the tool result should expose `page_id` to the model.** Currently no — it's noise unless we add a follow-up tool that takes a page_id. Revisit with `fetch_article` decision.
