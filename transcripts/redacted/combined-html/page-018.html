This document captures the design narrative for the Claude + Wikipedia QA

## Eval-set validation

The deliberate inclusion of non-search categories surfaced the headline v1
defect described below. A correctness-only eval focused on search-and-retrieve
(simple factual lookups, multi-hop chains, disambiguation) would have missed
it entirely. Hand-curated stress tests with explicit failure-mode coverage —
`negative_capability`, `unanswerable_not_in_wp`, `unanswerable_too_recent` —
produced signal that random sampling from a public QA dataset (HotpotQA,
SimpleQA, TriviaQA) would not have produced.

This is a vindication of the eval-design choice captured in DECISIONS at
2026-05-03 15:57: "stress-test specific failure modes deterministically"
over "maximize topical coverage." Each category is a hypothesis about a
failure mode; the categories that look low-yield (will the model search for
arithmetic? will it refuse appropriately on real-time questions?) turned out
to be the categories that caught the most consequential prompt defect.

## v1 baseline finding: prompt format defect on non-search cases

The v1 baseline run (34 cases, all 10 categories, Opus 4.7 for both agent and
judge) surfaced a structural defect in the v1 prompt.

The v1 prompt's output format assumed all answers were grounded in retrieved
content. It instructed the model to emit:

```
<evidence>...quoted passages from retrieved articles...</evidence>
<answer>...prose answer with inline citations and a Sources: section...</answer>
```

On `negative_capability` cases (math, code, opinion) and `unanswerable_*`
cases (real-time weather, recent NBA scores), the model correctly didn't
search Wikipedia and produced clean, useful prose responses — for arithmetic
it computed "1247 × 393 = 490,071" with a clear "not searching, this is
calculation" rationale; for unanswerable real-time questions it produced
clean refusals explaining Wikipedia is the wrong source and suggesting
alternatives. But without the `<evidence>/<answer>` wrappers, the parser
couldn't extract content (`evidence=""`, `answer=""`) and the judge scored
zero on factual_accuracy, groundedness, and citation_quality.

**The agent's behavior was correct; the prompt didn't tell it what format to
use when the grounding assumption didn't apply.** v1.1 generalizes the
output structure to require the wrapper even on non-search responses (e.g.,
`<evidence>none — Wikipedia is not the appropriate source for this
question</evidence>`).

This is the kind of finding the eval was designed to surface: not a model
weakness, but a prompt-engineering oversight that only becomes visible when
the system is exercised across the full distribution of user intents,
including ones it shouldn't be ground-truth-answering.

## Approach summary (one-pass)

- **Single tool, `search_wikipedia(query: str)`.** Per the brief.
- **Default-on search framing.** Tool description tells the model to search
  for any verifiable claim; exceptions explicitly enumerated (arithmetic,
  code, opinion, conversational content). Stronger than a permissive "use
  when needed" formulation that left too much room for the model to skip
  search and answer from priors.
- **XML output structure** (`<evidence>` then `<answer>`). Models attend
  well to XML; the structure makes evidence auditable for the judge.
- **Strict parsing on order, tolerant parsing on content.** Reversed-order
  blocks (answer before evidence) are treated as malformed — that ordering
  suggests post-hoc rationalization. Unescaped `<>&` inside extracts is
  tolerated (substring extraction, not `xml.etree`) so legitimate quoted
  content doesn't break parsing.
- **Search budget cap of 5 calls.** Soft fail — agent should answer with
  what it has rather than running out the clock.
- **Surface premise discrepancies, don't correct.** "Wikipedia indicates X;
  the question's premise was Y" rather than "you're wrong." Driven by
  customer-tone considerations.
- **Pydantic contracts at workstream boundaries** (`agent_contract.py`).
  Validation at construction; JSON round-trip; `ParseWarning` taxonomy as a
  StrEnum for categorical structural-failure signals.

84

87	search and answer from priors — defeating the entire point of a

100	groundedness and citation_quality inspectable, both for humans and the

102	compare claims in the answer to passages in evidence rather than

109	ability to handle synthesis questions; without it, the prompt would
110	paralyze the model on any question requiring connecting two facts.

113

136	for the photoelectric effect, not relativity." That was reversed
137	mid-design on customer-tone grounds: the model is not the arbiter of
138	truth and shouldn't position itself as correcting the user. Its job is
139	to make the disagreement legible — "Wikipedia indicates X; the
140	question's premise was Y" — and let the user reconcile it.
141	
142	This shows up in three places: the rubric's `calibration` dimension
143	(which now penalizes assertive correction even if the fact is right),
144	the dataset flag (`must_surface_premise_discrepancy`), and the system
145	prompt's edge-cases section.
146	
147	### Decomposition over compound queries; 5-call budget
148	
149	For multi-part questions, the prompt instructs the agent to decompose
150	into separate searches (one per facet) rather than issue compound
151	queries. Hypothesis: MediaWiki search rewards specific terms; long
152	compound queries dilute the signal. The trade-off — extra searches
153	that don't help — is a thing eval should flag if it's happening; that's
154	what `search_efficiency` is for.
155	
156	The search budget is 5 calls per question, with the explicit
157	instruction "answer with what you have rather than running out the
158	clock." This makes the cap a soft fail: the model should triage rather
159	than panic. 5 is a starting number; if eval shows multi-hop questions
160	hitting the cap, raise; if shows wasted searches near the cap, lower.
161	
162	### Reflection between searches
163	
164	After each tool call, the prompt requires the agent to make an explicit
165	decision: stop, refine the query, decompose to a different facet, or
166	admit Wikipedia doesn't have it. Four explicit branches make the
167	decision tree legible to the model. This was firmed up after observing
168	that Opus 4.7 sometimes gives up after one weak result.
169	
170	### Length: 2-4 paragraphs
171	
172	"Thorough but not comprehensive." Not a Wikipedia dump; a focused
173	answer. Single-fact questions should be short. The qualifier "longer
174	for genuine synthesis" preserves room for multi-source questions to
175	get the treatment they need.
176	
177	## Eval design
178	
179	### Hand-curated stress tests, not stratified random sampling
180	
181	The eval set is ~35 cases across 10 failure-mode categories rather
182	than a sample from a public dataset like HotpotQA, SimpleQA, or
183	TriviaQA. The taxonomy, per-category counts, difficulty mix, and
184	`expected_behavior` flag semantics are mine; an agent drafted
185	individual cases (questions, gold answers, notes) within those
186	constraints, and I reviewed before they entered the dataset. See
187	"Use of AI in development" below for a fuller account of how the
188	agent was directed and what review I applied.
189	
190	Trade-off accepted: no statistical confidence on average performance,
191	only pattern detection. With most categories at 2–4 cases, the eval
192	can detect meaningful regressions and surface failure patterns, but
193	can't put confidence intervals on per-dimension means.
194	
195	What I gain in exchange: signal density. Every case is a hypothesis
196	about a specific failure mode. Public-dataset random samples are
197	mostly easy and don't stress the behaviors that drive
198	prompt-engineering decisions. The curated set lets every case inform
199	the rubric and every failure point at a specific fix.
200	
201	I considered adversarial generation (red-team Claude to produce hard
202	cases). Rejected for v1 because it adds setup time and yields cases
203	without curation; I'd rather have 35 deliberate cases than 100
204	unvalidated ones.
205	
206	### 10 failure-mode categories
207	
208	Each category targets a distinct behavior the system might get wrong:
209	
210	| Category | What it tests |
211	|---|---|
212	| `simple_factual` | baseline lookup |
213	| `multi_hop` | sequential A→B→C reasoning; decomposition; knowing when to stop |
214	| `multi_source` | parallel info gathering across articles (covers comparative + synthesis) |
215	| `disambiguation_explicit` | search returns a Wikipedia disambiguation page; pick a sense and note alternatives |
216	| `buried_answer` | answer exists in the article but not in the lead extract |
217	| `negative_capability` | should NOT search (math, code, opinion) |
218	| `false_premise` | factually wrong premise embedded in question; surface the discrepancy |
219	| `unanswerable_not_in_wp` | answerable in principle but not in Wikipedia; refuse cleanly |
220	| `unanswerable_too_recent` | events past Wikipedia's freshness; cutoff caveat |
221	| `temporal` | recency-bounded; caveat with last known state |
222	
223	Categories were honest about their statistical power. Iteration-signal
224	categories (4+ cases) are `simple_factual`, `multi_hop`,
225	`negative_capability`, `false_premise`. Pattern-detection categories
226	(3 cases) are `multi_source`, `disambiguation_explicit`,
227	`buried_answer`, `unanswerable_not_in_wp`. Alarm-only categories
228	(2 cases) are `unanswerable_too_recent`, `temporal` — a single failure
229	means look closer; a pass is inconclusive.
230	
231	### Per-dimension rubric, no cross-dimension aggregation
232	
233	The judge scores five dimensions 0–3:
234	
235	- **factual_accuracy** — does the answer match what's true
236	- **groundedness** — are claims supported by retrieved content
237	- **citation_quality** — are citations precise and correctly attributed
238	- **search_efficiency** — were searches well-targeted, not wasteful
239	- **calibration** — does the model know what it knows; refuse when
240	  appropriate; surface premise discrepancies descriptively; express
241	  uncertainty when warranted
242	
243	Aggregation rule: **no aggregation across dimensions within a single
244	case.** Averaging factual_accuracy and citation_quality together is
245	meaningless and obscures failures. **Per-dimension means across cases
246	per iteration are computed and reported** — this is how I measure
247	iteration deltas, e.g. "v2 lifted factual_accuracy from 2.1 → 2.6 but
248	groundedness dropped from 2.5 → 1.8."
249	
250	`groundedness` was added explicitly to separate it from
251	`factual_accuracy`. An answer can be factually correct but ungrounded
252	— the model knew it from priors, not from the retrieved content.
253	That's a fail on groundedness, pass on factual_accuracy. Without the
254	separation, this failure mode is invisible.
255	
256	### Deterministic checks alongside the judge
257	
258	Some failure modes don't need an LLM. Eleven checks run alongside the
259	judge and produce a separate `behavior_checks` block in the result, so
260	hard-fails aren't conflated with rubric scores:
261	
262	- `searched_when_required`, `did_not_search_when_prohibited`,
263	  `not_excessive_searches`, `answer_length_plausible`
264	- `has_bracket_citations`, `no_markdown_links` (split — different
265	  failure modes, different fixes)
266	- `has_collated_sources` (Title - URL plain-text format)
267	- Four parse-warning cluster checks consuming the `ParseWarning` enum
268	  emitted by the response parser:
269	  - `output_has_required_blocks` (missing-block: model didn't emit
270	    structure → strengthen prompt)
271	  - `output_blocks_well_formed` (unclosed-tag: tried, malformed →
272	    tokenization/length investigation)
273	  - `output_blocks_non_empty` (empty-block: structure clean, content
274	    missing → prompt requirement to populate)
275	  - `output_blocks_canonical` (reversed-order or multiple-blocks: odd
276	    emission → emphasize evidence-first single-block)
277	
278	The cluster split was a deliberate choice. An earlier draft collapsed
279	all 9 parse-warning codes into one binary `no_parse_warnings` check.
280	Reversed because the codes map to genuinely different fixes, and
281	collapsing them hides which one is firing.
282	
283	### Judge sees a deliberate subset
284	
285	The judge prompt is built from a `JudgeInput` Pydantic model — an
286	explicit data shape that makes "what we send the judge" readable from
287	one source.
288	
289	Included: `question`, `expected_answer`, `expected_behavior` (the
290	judge-context flags `must_surface_premise_discrepancy` and
291	`must_refuse`), parsed `evidence`, parsed `answer`, `parse_warnings`,
292	and `tool_calls` (each with `query` + `raw_result_str` + `latency_ms`,
293	so the judge can assess groundedness against what the agent actually
294	saw).
295	
296	Excluded: `raw_output` (we already have parsed `evidence` and `answer`
297	— sending the unparsed model text would just bloat the prompt),
298	`raw_messages` (the full conversation including the system prompt —
299	the same across cases, redundant), and convenience fields (`queries`,
300	`n_searches`, `stop_reason`, `usage`) that duplicate `tool_calls` or
301	aren't relevant to the judge's task.
302	
303	`parse_warnings` is included as **informational context, not a scoring
304	directive.** The prompt says use them to *interpret* the answer ("an
305	unsupported claim alongside `empty_evidence_block` reads as the model
306	failing to populate evidence rather than asserting something
307	ungrounded"), but explicitly does NOT apply additional rubric
308	penalties on this basis — the harness already records structural
309	failure separately via the four parse-warning behavior checks. Without
310	this guidance the judge would double-count.
311	
312	### Judge robustness: parse, retry, isolate
313	
314	The judge produces XML in a fixed schema (one `<dimension>` per rubric
315	dimension, `<reasoning>` before `<score>`). Reasoning before score is
316	per Anthropic's prompt-engineering guidance — separating reasoning
317	from final output via structured tags improves calibration on judgment
318	tasks.
319	
320	Parser handling:
321	- Missing dimension → that dim's score = `None`, flagged; other dims
322	  still score
323	- Score out of range → clamp to 0–3 + flag
324	- Malformed XML on first attempt → retry once with explicit "return
325	  only the `<evaluation>` block" guidance
326	- Second malformed → mark `judge_failure=true`, exclude from per-dim
327	  means, surface count separately
328	
329	Per-dim means are computed only over cases where the judge produced
330	valid output. Per-dim `judge_failure_rate` is its own metric.
331	
332	### Calibration
333	
334	The LLM judge needs validation against human judgment to be
335	trustworthy. The plan: sample 5–10 cases stratified across dimensions
336	and judge-score buckets, dump them to a markdown sheet for human
337	review, fill in human scores via a YAML round-trip, and compute per-dim
338	agreement (`|human - judge| ≤ 1`). If a dimension shows >25%
339	disagreement, that's signal to revise the rubric or judge prompt.
340	
341	YAML for the human input rather than markdown round-trip: parsing
342	human-edited markdown is fragile (whitespace, partial fills, score
343	formats). YAML keeps the input strict and the read-only markdown view
344	separate.
345	
346	## What this eval measures — and what it doesn't
347	
348	The v1 eval measures correctness: factual_accuracy, groundedness,
349	citation_quality, search_efficiency, calibration. These are the
350	dimensions that determine whether the answer is right. They don't
351	measure voice or tone — whether uncertainty is expressed with
352	confidence or apologetically, whether surfacing a false premise feels
353	respectful or corrective, whether the agent's prose has the warmth
354	that distinguishes a thoughtful response from a merely competent one.
355	This gap is real and deliberate. Voice failures don't surface cleanly
356	in Wikipedia QA categories — they live in conversational shapes
357	(frustrated users, ambiguous emotional contexts, situations requiring
358	graceful uncertainty) that aren't represented in the dataset. Adding a
359	voice dimension would require both case expansion and judge
360	calibration against human-scored examples, since voice scoring is
361	more subjective than correctness scoring and the LLM judge would need
362	a more calibrated rubric to score it reliably. v2 would address this.
363	For v1, the calibration dimension partially captures voice-adjacent
364	behavior — "surface the discrepancy without correcting" is partly a
365	voice question — but the rubric isn't designed to score voice as a
366	first-class concern.

395	
396	- **Buried-answer recovery.** Under the single-tool design with
397	  `exintro=true`, the lead extract is the only article content the
398	  agent ever sees. Bumping `exchars` widens the lead view but never
399	  reaches into the article body. For `buried_answer` cases, recovery
400	  is solely via re-searching with more specific queries that surface a

402	  meaningful failure rate on this category, the structural fixes are:
403	  (a) add `fetch_wikipedia_article(title)` as a second tool, or

### Structural finding: prompt format defect on non-search cases

**v1 baseline surfaced a prompt defect.** The v1 output format assumed all
answers were grounded in retrieved content. On `negative_capability` and
`unanswerable_*` cases, the model correctly didn't search and produced
clean prose — but without `<evidence>` / `<answer>` wrappers, the parser
couldn't extract content and the judge scored 0. The agent's behavior was

assumption didn't apply. v1.1 generalizes the output structure.

parse warnings — every `negative_capability` and `unanswerable_*` case.

factual_accuracy / groundedness / citation_quality.

most consequential prompt defect. A random-sampled QA dataset would
have indexed on factual lookups and missed this entirely.

You are a research assistant. You answer user questions by searching English Wikipedia and grounding your answers in the content you retrieve. Your value comes from giving verifiable, well-sourced answers — not from showcasing broad knowledge you may have memorized.

## The grounding rule

Every factual claim in your answer must trace to a quoted passage in your `<evidence>` block. **Your evidence block is the authoritative surface for what may appear in your answer.** If it is not in evidence, do not claim it — even if you are confident from training. This is what protects the user's ability to verify what they are being told.

If a chain of reasoning leads you to a conclusion that the evidence does not directly state, you may share it — but you must mark it explicitly: "Wikipedia says X; from this it follows that Y, because [reason]."

## Searching Wikipedia

Default to using `search_wikipedia` whenever your answer would benefit from being verifiable: facts about people, places, events, dates, definitions, technical concepts, history, comparisons. When in doubt, search.

Skip the tool when Wikipedia is the wrong source:
- Pure arithmetic or calculation
- Code generation
- Opinion, preference, or creative writing ("write me a poem about X")
- Questions about content the user has already given you in this conversation

For multi-part or comparative questions, search each component separately rather than constructing a compound query. "Impact of the Gulf War on oil prices" is two queries (Gulf War causes/timeline; oil prices in that period), not one.

Favor short, specific noun-phrase queries over conversational rephrasing. "Treaty of Versailles signatories" beats "who signed the Treaty of Versailles?". MediaWiki search rewards specific terms.

You have a budget of **5 search calls per question**. Stay within it. If you find yourself running out without converging, answer with what you have (and name what you could not verify) rather than running out the clock.

After each search, pause and assess: do these results contain what I need to answer the question?
- If yes: stop searching and produce the answer.
- If no, and a better query is now obvious: refine and search again.
- If no, and the question has another part you have not searched yet: search the next part.
- If multiple searches do not surface the information: stop, and say so in the answer rather than guessing.

## Disambiguation

If the user has specified which sense they mean ("Java, the programming language") respect that — search for that sense and do not waste a query on alternatives.

If the term is ambiguous and the user did not specify, pick the most plausible reading given the surrounding context, but acknowledge the alternatives in your answer ("Java most commonly refers to the programming language; the island and the coffee are separate topics with their own articles.").

## Producing the answer

**You always emit both `<evidence>` and `<answer>` blocks**, regardless of whether you searched Wikipedia. The wrapper structure is required on every response.

**For grounded answers** (you searched and retrieved content), the evidence block contains quoted passages from the articles you retrieved:

[Source: Albert Einstein] "Einstein was awarded the 1921 Nobel Prize in Physics for his discovery of the law of the photoelectric effect."

</evidence>

His relativity theories — special, published in 1905, and general, published in 1915 — were not the basis for the prize, despite being the work he is most popularly associated with.

Theory of relativity - https://en.wikipedia.org/wiki/Theory_of_relativity
</answer>

</evidence>
